## 1. Root cause of the mismatch

The Today panel and push notifications use **two different suppression stores with different lifetimes**, and they drift apart:

| | Where | Lifetime | Controls |
|---|---|---|---|
| `iw_snoozed` | browser `localStorage` (`index.html:448`) | **permanent, never pruned** | what the app *shows* |
| server dismissals | `KVSetting` per user (`recommendations.py`) | **7-day TTL** (`_DISMISS_TTL_DAYS`) | what push *suppresses* |

Flow that produces the bug:
1. User taps **Accept** or **Ignore** on a card ŌåÆ id is written to `iw_snoozed` forever (`index.html:672,677`) **and** to the 7-day server dismissal.
2. After 7 days the server dismissal expires ŌåÆ the recommendation resurfaces server-side ŌåÆ `evaluate_and_notify` / the digest **push it again**.
3. The app still filters it out via the permanent `iw_snoozed` list ŌåÆ **"Nothing to do right now."**

The Ōé¬414/month contribution card is not missing ŌĆö it is generated by `_behind_goal_recs` (`recommendations.py`), included in `/api/v1/recommendations`, and referenced by the digest via `context_service.gather`. It is simply being hidden by the stale client-side list.

### Secondary gaps that widen the mismatch
- **Informational pushes look actionable.** Price-move ("META moved +10.2% ŌĆö No action needed") and the weekly digest render in the same notification stream, same styling as real to-dos. Users read them as "an action."
- **Pushed risk alerts have no in-app card.** `evaluate_and_notify` pushes `ŌÜĀ’ĖÅ Risk alert` for any concentration vector (`geographic`, `currency`, `liquidity`, ŌĆ”), but only `single_position` concentration produces an actionable Today card (the "Trim" rec). A pushed geographic-concentration alert maps to nothing you can act on in "What to do now" ŌĆö it only appears in the separate "Your biggest risk" panel.
- **Two projections disagree.** `_behind_goal_recs` sizes the gap with a deterministic fixed-return projection (digest showed ~Ōé¬25,653), while the home "Where this could end up" panel uses the Monte-Carlo `/api/v1/plan/projection` (median ~Ōé¬22,135). The contribution number in the notification therefore won't match the app's own projection.

## 2. Fix design

### A. One source of truth for "dismissed" (kills the primary bug)
- **Remove the client-side `iw_snoozed` filter as the authority.** The server dismissal (7-day TTL) already drives both push and `/api/v1/recommendations`; make the app trust the server list instead of a permanent local one.
  - Delete the `.filter(o=>!snoozed.includes(o.id))` gate in `loadToday` (`index.html:599`); render exactly what the endpoint returns.
  - Keep a *short-lived optimistic* local hide only to make Accept/Ignore feel instant, but expire it (e.g. store `{id: dismissedAtISO}` and drop entries older than the same 7-day TTL) so it can never outlive the server. This makes app and push provably consistent.
- **Net effect:** if a card can notify, it shows in the app; if it's dismissed, both go quiet; when it resurfaces after 7 days, it resurfaces in *both* places together.

### B. Bucket notifications: actionable vs informational (two-way sync)
Give every push a `category` so the client and the wording stay honest:
- `action` ŌåÆ maps 1:1 to a Today card (recommendations, triggered rules, and ŌĆö see C ŌĆö actionable risk alerts). Title stays "­¤ÆĪ ŌĆ”".
- `info` ŌåÆ price moves and the digest. Title/wording made explicitly non-actionable ("­¤ōł FYI ŌĆö META +10.2%, no action needed"); tapping opens the relevant read-only view, not the to-do list.
- Enforce the contract in `evaluate_and_notify`: only push `action` items whose id is currently in `build_recommendations(...)`. This guarantees an actionable push always has a matching card.

### C. Give pushed risk alerts a real to-do (or stop implying action)
- Add actionable diversification cards for `geographic` and `currency` concentration in `build_recommendations`, mirroring the existing single-position "Trim" card (why + how + impact + `apply` spec where a sensible action exists).
- Where no concrete trade is appropriate, downgrade that alert's push to `info` so it doesn't imply a to-do. Either way, every `ŌÜĀ’ĖÅ`-style actionable push now corresponds to a card.

### D. Make the goal-gap contribution a first-class, consistent to-do
- Keep the `_behind_goal_recs` contribution card, but **source its projection from the same engine as the home panel** (Monte-Carlo `/api/v1/plan/projection` median) so the gap and the Ōé¬/month figure match what the app shows. One projection, used by the card, the digest, and the home panel.
- Ensure its `apply.kind` (currently `"none"`) degrades gracefully ŌĆö Ignore should snooze it for the TTL, not error, and it should never be permanently hidden by the old local list.

### E. Recommendations: "why + how + impact" (accuracy + clarity)
Add two structured fields to each rec and render them:
- `why` ŌĆö the trigger in plain language ("Your VTI is 34% of the portfolio, above your 25% cap").
- `impact` ŌĆö the expected effect, quantified from the existing engines ("Brings single-name risk to your 25% limit"; "Closes ~60% of your Ōé¬31k goal gap"; "Saves ~Ōé¬X in tax this year").
- Keep `action` (what) and `how` (steps) and the existing `audit_trail` accordion (already strong for accuracy/traceability).
- Accuracy pass: reconcile the projection engines (D), and label each card's freshness using the existing data-source status so numbers are never silently stale.
